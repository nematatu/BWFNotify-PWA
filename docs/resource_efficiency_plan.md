# Cloudflare 資源効率化・大規模ユーザー向け最適化プラン

Cloudflare Worker、KV、Web Pushの無駄な呼び出しと課金リスクを極小化しつつ、リアルタイム性を担保する設計変更プランです。既に実装・検証を完了しています。

---

## 🛠️ 3つのコア最適化設計

### 1. 🗄️ KV読み書きの極小化 (Metadata-driven Subscriptions)
*   **従来の課題**: 
    *   プッシュ通知を送る際、`kv.list()` ですべての購読キーを取得し、その後**ユーザー数と同じ回数の `kv.get()`** を実行していました。ユーザーが1万人いると、通知を送るたびに1万回のKV読み取りが発生し、一瞬で無料枠（1日10万回）を使い切り高額請求につながります。
    *   ユーザーが画面を開くたびに、購読確認のための KV 書き込みが走っていました。
*   **今回の改善策**:
    *   **KVメタデータの活用**: ユーザー購読データ（`keys`, `endpoint`, `excludedMatchIds`）を、KVの**メタデータ**（1024バイト制限）に書き込むよう `saveSubscription` と `updateSubscriptionExclusions` を変更しました（不要な `userAgent` は除外して容量を節約）。
    *   **一括リスト取得の実現**: 通知送信時は `kv.list<{ metadata }>()` を呼び出し、リスト内の `metadata` から購読情報を直接取り出すことで、**`kv.get` を完全にゼロ（0回）に削減**しました。1万人のユーザーがいても、KVのリスト呼び出し数回（1ページ1,000件）だけで済み、**読み取り回数が1,000分の1以下**になります。
    *   **冗長な書き込みの排除**: 送信された購読情報が既存のレコードと完全に一致している場合は `kv.put` をスキップする論理チェックを導入し、無駄な書き込み回数を削減しました。

### 2. ⏳ クライアントサイドのインテリジェント・ポーリング (Conditional Live Poll)
*   **従来の課題**:
    *   画面が開いている間、試合が「ライブ中」か「予定」に関わらず、15秒に1回のペースで `/api/live` を常にポーリングし続けていました。
*   **今回の改善策**:
    *   **予定のみの時間は静かに待機**: 画面上に「ライブ中」の試合が1つもないときは、15秒ごとの `/api/live` への高頻度ポーリングを**完全に停止（タイマーを解除）**します。
    *   **更新は全体同期のみ**: 2分に1回の全体更新（`/api/status`）のみを実行し、そのレスポンスの中に「ライブ中」の試合が検出された場合のみ、15秒ごとの高頻度ポーリングを動的に再開します。ライブ試合が終了したら、再び自動で高頻度ポーリングを停止します。

### 3. 💤 ユーザー不活性（Idle）検知による自動スリープ
*   **従来の課題**:
    *   PCやスマートフォンのブラウザタブが開いたまま放置されると、ユーザーが画面を見ていないにも関わらず、永久にAPIへアクセスし続けていました。
*   **今回の改善策**:
    *   **5分無操作でスリープ**: マウス移動、キー入力、画面スクロール、タップなどの操作が5分間全くない場合、ユーザーが離脱したと判断して**すべてのポーリングを完全に停止**します。
    *   **復帰時に即座に再開**: ユーザーが画面を再びスクロールしたりタップしたりした瞬間、スリープを解除して即座に最新状態を再取得し、定期ポーリングを再開します。

## 📈 削減効果の見込み (1万ユーザー想定)

| 項目 | 対策前 | 対策後 | 削減率 / 効果 |
|---|---|---|---|
| **通知配信時のKV読み取り** | 10,000回 / 配信 | **10回** / 配信 | **99.9% 削減** |
| **無駄なライブスコア要求** | 2,400,000回 / 日 | **ほぼ0回** (ライブ試合のない時間帯) | **95%以上 削減** |
| **放置タブのAPI要求** | 無制限に発生 | **5分で完全停止** | 大幅な要求数削減 |
| **不要な購読書き込み** | 開くたびに毎回発生 | **差分発生時のみ** (初回/変更時) | **90%以上 削減** |
