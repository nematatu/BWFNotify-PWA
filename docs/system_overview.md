# システム概要・引継ぎドキュメント (System Overview & Handover)

このドキュメントは、このシステムの全体像、アーキテクチャ、設計思想、解決済みの課題、および今後の展望をまとめたものです。
**開発や仕様変更を行うAIおよび開発者は、本ドキュメントを変更の度に更新し、常に最新のシステム状況を維持してください。**

---

## 1. システム全体像 (The Big Picture)

BWFNotify-PWA は、バドミントン世界連盟（BWF）の公式大会において、**「日本人選手の試合情報」をリアルタイムに抽出し、試合開始をモバイル・デスクトップへプッシュ通知し、YouTube公式配信リンクを提供**するWebアプリケーション（PWA）です。

```
                    [ BWF Badminton Extranet API ] (2分毎にデータ取得)
                                  │
                                  ▼
[ Cloudflare Workers (Hono) ] ─── [ Cloudflare KV (NOTIFIED_MATCHES) ]
  (バックエンドロジック・Cron実行)           (最新ステータス & 購読情報保存)
        │                                 │
        ├─────────────────────────────────┘
        ▼
[ Web Push (FCM / APNs) ] ──────► [ iOS / Android / Desktop (PWA) ]
  (試合開始をプッシュ通知)             (Vanilla JS + CSS フロントエンド)
                                        - 15秒間隔のライブ得点表示
                                        - YouTube公式配信リンクへの誘導
                                        - 5分無操作スリープ（不活性検知）
```

---

## 2. アーキテクチャと使用サービス

### フロントエンド (SPA)
- **構成**: HTML5、Vanilla JavaScript、Vanilla CSS（CSSフレームワーク不使用）。
- **特徴**: ライトテーマ固定（ダークモードなし）、フォントは `LINE Seed JP`。
- **PWA / Service Worker**: `public/pwa/sw.js` がオフライン対応、キャッシュ、および Web Push の受け取りと表示を担当。iOS Safari のホーム画面追加（PWA）におけるプッシュ受信に対応。

### バックエンド (Serverless)
- **環境**: Cloudflare Workers + Hono フレームワーク。
- **データ永続化**: Cloudflare KV（Namespace: `NOTIFIED_MATCHES`）。
- **Cronスケジュール**: `*/2 * * * *`（2分に1回起動）。BWF APIから情報取得、ライブ判定、通知送信用ペイロード作成、状態更新を実施。

## 3. 目指すことと設計思想

1.  **ランニングコスト $0 の維持**
    - 全体のアクセス数がどれだけ増加しても、Cloudflareの無料枠（リクエスト数10万回/日、KV読み取り10万回/日、書き込み1000回/日）を絶対に超過しないよう、クライアント・サーバー双方で極限までリソース消費を抑える設計を適用します。
2.  **日本人選手ファースト**
    - 取得した対戦カードを自動で正規化し、日本人選手を必ず左側（先頭）に配置します。名前を大きく表示しマーカー下線で強調します。
3.  **公式情報の厳密なマッチング**
    - 選手名・対戦コート・日付などを組み合わせ、YouTube公式配信元（BWF TVや地域連盟など）の配信URLをサーバー側で厳密に特定します。非公式リンクや推測リンクは絶対に提供しません。

## 4. 解決済みの主要な課題 (Solved Challenges)

### 課金防止: KV読み取り回数の 99.9% 削減
- **課題**: 以前はプッシュ通知送信のたびに、全ユーザーのキーに対し個別に `kv.get` を実行していたため、ユーザー数に比例してKV読み取りが発生し、一瞬で無料枠を超過するリスクがありました。
- **解決策**: 購読情報（keys, endpoint, exclusions）を KV の **メタデータ（Metadata）** に同時に格納するよう変更。送信時は `kv.list<{ metadata }>()` を呼び出して一括で情報を受け取ることで、個別の `kv.get` を完全に0回へ削減しました。

### 通信量削減: 動的ポーリング ＆ 不活性スリープ
- **課題**: 画面を開いたまま放置されたタブが永久にAPIをポーリングし、サーバーやBWF APIへ負荷をかけ続けていました。
- **解決策**:
    - **ライブ中以外はスリープ**: 画面上にライブ試合が存在しない場合、15秒間隔の得点取得（`/api/live`）を動的に停止し、2分に1回の全体同期のみで待機します。
    - **5分無操作スリープ**: マウス移動やスクロールなどのユーザー操作が5分間検知されない場合、すべての通信を完全停止します。操作を再開した瞬間にスリープを解除します。

### モバイル通知の強化
- **課題**: iOSなどのモバイル端末では、通知の `image` パラメータがサポートされておらず、選手写真が通知に出ない問題がありました。
- **解決策**: ペイロードの `icon` フィールドにも選手写真を渡し、`sw.js` 側で `payload.icon` を使ってアイコン枠にフォールバック表示させる仕組みを実装しました。

### TypeScript 7.0.2 のネイティブGo製コンパイラへの移行
- コンパイラを Go 言語で書かれた超高速な **TypeScript 7.0.2** へ移行。型チェック時間を **0.55秒 ➡ 0.11秒（約5倍高速化）** に短縮し、開発体験とCIの高速化を両立させました。

### 初回流入時のUX改善とアプリ内ブラウザ（WebView）対応
- **課題**: SNS（XやYouTubeなど）から流入した際、アプリ内ブラウザ（in-app WebView）ではPWAのインストールが技術的に不可能であるにも関わらず、以前は全画面型のポップアップでインストールを強制していたため、アプリの価値が伝わる前に離脱される高いリスクがありました。
- **解決策**:
  - **非遮蔽ガイド**: 初回の画面占有型ポップアップを完全に廃止し、上部に情報を示すバナー（`pwa-guide-banner`）を配置。ユーザーは試合スコアや公式配信などの提供価値をまず自由に確認できます。
  - **アプリ内ブラウザ自動検知**: WebViewからのアクセスを `isInAppBrowser()` で検出。自動的に「通知を受けるには標準ブラウザ（SafariやChrome）で開き直してください」と案内する親切なメッセージに切り替わるようにしました。

## 5. 今後の課題と技術的注意点

1.  **D1データベースへの完全移行検討**
    - 現在はKVメタデータで購読一覧を読み出していますが、さらに規模が大きくなった場合（数万人規模）、リレーショナルでより軽量なインデックス検索が可能な **Cloudflare D1 (無料枠 500万リード/日)** への移行が、パフォーマンスとコスト（完全に無料枠内）の観点で次の選択肢になります。
2.  **キャッシュライフサイクルの検証**
    - 現在 `/api/live` は 10秒、`/api/status` は 30秒 Edge キャッシュされています。試合の進行が激しい時間帯におけるユーザー間のスコア同期のズレ具合に注意してください。
3.  **YouTube配信リンクの厳密性維持**
    - `src/config/youtube-stream-sources.ts` に設定された公式チャンネルから特定する仕組みですが、大会名表記のブレ（大文字小文字、国名）に柔軟に対応できるようにテストコードのメンテナンスを継続してください。
