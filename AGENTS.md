# AGENTS.md

このファイルは、このリポジトリで作業する開発者およびコーディングエージェント向けの必須ルールです。会話中の一時的な判断より、このファイルと利用者の最新指示を優先してください。

## 1. リポジトリの境界

- このリポジトリは `BWFNotify-PWA` 専用です。CLIは別リポジトリで管理するため、CLIのコード、コマンド、パッケージ、READMEを追加しないでください。
- 旧リポジトリや旧Workerは参照元にすぎません。実装を参照する場合でも、旧リポジトリ、旧Worker、旧KV、旧ドメインを変更、削除、再デプロイしないでください。
- 対象Workerは `bwfnotify-pwa`、公開先は `https://bwfnotify-pwa.amtt.workers.dev/` です。他のWorkerへ操作を行わないでください。
- 利用者の既存変更や無関係な変更を取り消さないでください。作業前後に `git status --short` と差分を確認してください。
- 長時間動作するローカルサーバーは、テストや目視確認に必要な場合だけ起動してください。作業後は必ず停止し、特にポート8787を残さないでください。

## 2. 事実確認と報告

- 利用者への返答は日本語の敬語を使用し、結論を先に簡潔に伝えてください。不自然な英語や内部用語で格好をつけず、一般的な日本語へ言い換えてください。
- 未確認の事実を断定しないでください。BWF、YouTube、Cloudflare、Web Push、ブラウザ対応状況は、コード、実レスポンス、公式資料のいずれかで確認してください。
- 「変更した」「戻した」「デプロイした」「通知できる」と報告する前に、現在の状態をコマンドまたは実動作で確認してください。
- 報告では「現在どうなっているか」を先に短く示してください。途中で行った操作の羅列だけで回答しないでください。
- 旧環境と現環境を曖昧に「旧」「新」と呼ばず、必要に応じてリポジトリ名、Worker名、URLを明記してください。
- 失敗を環境要因と判断する場合も、エラー内容を確認してから説明してください。

## 3. 作業管理

- 複数の修正点がある作業では、実装前に漏れのないチェックリストを作成してください。
- 作業中はチェックリストを更新し、各段階で未完了項目が残っていないか再確認してください。
- 最新の利用者指示を優先し、古い要件との競合を解消してから編集してください。
- 変更対象、変更しない対象、検証方法を編集前に明確にしてください。
- 要件を一部だけ実装して完了扱いにしないでください。完了条件をすべて検証できない場合は、未確認項目を明記してください。

## 4. UIの必須方針

- 常設画面はライトテーマとし、ダークモードを追加しないでください。赤いブランドヘッダー以外は白・薄いグレーを基本にしてください。
- 画面フォントは `LINE Seed JP` を第一候補にしてください。フォント配信元のCSP設定とフォールバックを維持してください。
- 装飾目的の角丸カード、四角形の一辺だけを着色したボーダー、グラデーション、過剰な影、意味のないバッジやアイコンを追加しないでください。
- カード、ボタン、選択欄は原則として直線的な境界を使ってください。円形の選手写真、トグル、件数表示など、形状に意味がある要素だけ丸形を使用できます。
- 操作状態を示す選択線、通知状態の緑、ライブ状態の赤など、意味のある差し色は維持してください。
- 不自然な英語ラベルを避け、利用者向け文言は簡潔な日本語にしてください。内部の値やAPI用語をそのまま画面へ出さないでください。
- 既存アイコンで表現できる操作に、文字入りの装飾ボタンや独自の図形を追加しないでください。外部リンクは一般的な外部リンクアイコンで示してください。
- 320px、390px、デスクトップ、広幅画面で、文字、国旗、選手画像、得点、操作欄が重ならないことを確認してください。
- 広幅画面を常に縦1列にしないでください。カード数に応じて複数列を使い、不要な縦スクロールを減らしてください。
- UI変更はPlaywrightのスクリーンショットを生成して目視確認してください。DOM検査だけで表示品質を判断しないでください。

## 5. 試合表示の要件

- 「ライブ中」と「このあと」は同じ試合領域の横並びタブで切り替えられるようにしてください。「予定」という冗長な状態ラベルは試合カードへ表示しないでください。
- 日本人選手のチームを必ず左側または先頭へ正規化してください。日本人選手名は外国人選手名より大きく表示し、マーカー状の下線で目立たせてください。
- 対戦は横方向に `vs` または得点を挟んで表示してください。外国人名が収まらない場合はレイアウトを広げず、省略記号で処理してください。
- 大会順では大会ごとにまとめ、グループ内を時刻順にしてください。時刻順では大会グループを優先せず、全大会の試合を時刻で並べてください。選択したソート方法と実際の並びを一致させてください。
- 大会ロゴまたは大会画像を十分な大きさで表示してください。BWFから取得できない画像を推測で生成しないでください。
- 選手写真、国旗、WRを表示してください。国旗は判別可能な大きさにし、冗長な国名は併記しないでください。
- H2Hと前回対戦は対戦表示の直下へまとめてください。独立カラムを作って選手画像や国旗を重複表示しないでください。
- 前回対戦結果は勝者だけを明確に表示し、`team1勝利` のような内部的な表現を使わないでください。現在の日本人先頭順と過去対戦のチーム順を正しく対応させてください。
- ライブ中は現在得点、各ゲーム得点、サーブ側を表示してください。サーブ側は `server` などの文字ではなくシャトルアイコンで示し、動きが苦手な利用者向けに `prefers-reduced-motion` を尊重してください。
- 得点は画面表示中にリアルタイム更新してください。ページが非表示の間は不要なポーリングを止めてください。
- 外部試合リンクまたは配信リンクは、その試合に対応する直接URLを確認できる場合だけ表示してください。

## 6. YouTube配信リンク

- 選手名一致だけで配信URLを決めないでください。配信タイトルが大会名、日付、ラウンド、コートだけの場合を扱ってください。
- 大会ごとの公式配信元は `src/config/` で管理してください。BWF TV、主催者、地域連盟など、確認済みの公式チャンネルだけを候補にしてください。
- 大会、現地日付、コート、選手名の利用可能な情報を組み合わせ、別日、別コート、非公式チャンネルを除外してください。
- BWFで配信されない大会を考慮してください。確認済みの直接URLがない場合はリンクを表示せず、YouTube検索結果への代替リンクを生成しないでください。
- 配信探索はサーバー側で共有実行し、利用者ごとに行わないでください。一度確定したURLは再利用し、毎回再検索しないでください。
- 未配信時は短い間隔で無制限に検索し続けず、負のキャッシュと再確認時刻を設けてください。

## 7. 通知

- 日本人選手の対象試合は、初期状態ですべて通知対象にしてください。利用者は試合ごとのトグルで不要な通知だけを除外できるようにしてください。
- モバイルを含め、試合開始通知はサーバー側の定期処理とWeb Pushで送信してください。画面を開いていることを通知条件にしないでください。
- 通知設定、通知除外、購読再保存の動作を壊さないでください。購読再保存時も除外設定を維持してください。
- テスト通知ボタンは表示だけでなく、実際のService Worker通知またはPush経路まで検証してください。
- iOS/iPadOSのWeb Push要件、ホーム画面PWA、許可操作の制約を考慮してください。対応ブラウザを単純なUser-Agent判定だけで「非対応」と断定しないでください。
- 通知画像には対象の日本人選手画像を優先し、利用できない場合だけPWAアイコンへフォールバックしてください。
- OGPのタイトル、説明、正規URL、1200x630画像、Twitterカードを維持してください。OGP画像、PWAアイコン、通知画像を変更した場合は、実ファイルの寸法と公開レスポンスを確認してください。

## 8. Cloudflareと資源効率

- KVへ毎ポーリング書き込まないでください。試合状態、得点、購読状態などが変化した場合だけ書き込み、必要なハートビートには十分長い間隔を設けてください。
- KVの読み書きを利用者数に比例させないでください。共有できる試合状態、H2H、配信URLはサーバー側でキャッシュしてください。
- ライブ得点と全試合情報で更新頻度を分けてください。ライブ得点は画面表示中だけ短い間隔で取得し、重い全更新やYouTube探索を同じ頻度で実行しないでください。
- Cron、API、クライアントの三重取得を避けてください。どの層が取得、保存、配信を担当するかを明確にしてください。
- URL確定後、未配信時、試合終了後の停止条件を実装してください。
- 効率化を行う際は、KV書き込み回数、外部API呼び出し回数、クライアントのポーリング条件をテストしてください。

## 9. ディレクトリと設定

- 責務を混在させず、既存の `view/`、`api/`、`test/`、`type/`、`pwa/`、`config/` の区分を維持してください。
- 日本人選手名の辞書は `src/config/japanese-player-names.ts` で管理してください。画面表示、検索用別名、YouTube照合の用途を混同しないでください。
- 大会別の公式配信元は `src/config/youtube-stream-sources.ts` で管理してください。
- 選手名辞書を追加したら、BWFデータから日本語表示へ反映されるテストを必ず追加してください。
- PWAのCSS、JavaScript、Service Workerを変更した場合は、HTMLのクエリ版、モジュールimport版、Service Workerのキャッシュ名とアセット版を同時に更新してください。

## 10. 必須の開発サイクル

1. **現状確認**: `git status --short`、関連コード、既存テスト、現在の公開先を確認します。
2. **要件分解**: 利用者の指摘を検査可能なチェックリストへ変換し、対象外も明記します。
3. **調査**: BWFや外部サービスの仕様は実レスポンスまたは公式資料で確認します。推測したURLやデータを実装しません。
4. **小さく実装**: 既存の責務境界と実装パターンを維持し、無関係なリファクタリングを混ぜません。
5. **回帰テスト追加**: 修正した不具合が戻らないテストを先に、または実装と同時に追加します。
6. **段階検査**: 変更箇所の単体テスト、結合テスト、Playwrightの順に実行します。
7. **表示確認**: UI変更では320px、390px、デスクトップ、広幅のスクリーンショットを目視します。重なり、はみ出し、並び順、文字省略、画像位置を確認します。
8. **コミット前検査**: `bun run precommit` を実行し、型、Wrangler型、Lint、単体、結合、Playwright、Worker dry-runをすべて通します。
9. **セルフレビュー**: `git diff --check` と全差分を読み、要件漏れ、不要な変更、セキュリティ、KV負荷、モバイル動作を確認します。
10. **履歴管理**: まとまりのある変更単位でコミットし、`main` へpushします。作業途中でも意味のある安定点では適宜コミットしてください。
11. **自動デプロイ確認**: `main` へのpushでCloudflareのGit連携がデプロイします。明示指示がない限り `wrangler deploy` を手動実行しないでください。
12. **公開確認**: GitHub CIとCloudflareの自動デプロイ完了後、公開HTML、CSS、Service Worker、対象APIを確認します。キャッシュ版も照合してください。
13. **終了確認**: `git status --short` が意図した状態であること、テスト用サーバーが停止していること、チェックリストが全件完了していることを確認します。

## 11. テストとレビューの完了条件

- `bun run test:unit`: ドメインロジック、辞書、並び順、通知、YouTube、KV効率を検証します。
- `bun run test:integration`: WorkerのAPI、セキュリティヘッダー、Push購読などの結合を検証します。
- `bun run test:layout`: モバイルとデスクトップのレイアウト、重なり、横スクロール、表示状態を検証します。
- `bun run precommit`: コミット前に必ず実行します。失敗を無視してコミットしないでください。
- UI変更ではPlaywright画像の目視確認が必須です。テストが通っただけで見た目の完了としないでください。
- レビューでは、機能の正しさだけでなく、UX、アクセシビリティ、モバイルWeb Push、KV消費、外部API負荷、キャッシュ更新、旧環境への誤操作を確認してください。
- テスト失敗を修正するために検証を弱めないでください。期待値が誤っている場合だけ、理由を確認した上で更新してください。

## 12. Gitとデプロイ

- コミット前に差分をレビューし、コミットメッセージは変更内容を具体的に示してください。
- `main` へのpush後はGitHub CIとCloudflare Git連携を確認してください。
- 手動デプロイ、旧Workerのデプロイ、Worker削除、KV削除は、利用者がその操作を明示した場合に限ります。
- デプロイ完了を確認する前に「公開済み」と報告しないでください。
- 公開確認用のクエリ版を更新した場合、HTML、CSS、JavaScript、Service Workerで同じ版を使用してください。

## 13. 作業終了時の報告

- 現在の状態を最初に説明してください。
- 変更したファイルや主要な動作を簡潔に説明してください。
- 実行して成功したテストを具体的に示してください。
- コミットID、push先、自動デプロイ結果を確認できた範囲で示してください。
- 未確認事項や失敗があれば省略せず明記してください。
- 旧Worker、CLI、KVなど、触れてはいけない対象へ操作していないことを必要に応じて明記してください。
